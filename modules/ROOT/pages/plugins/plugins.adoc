= Plugins

== Introduction
A plugin is a technical extension designed to extend the core capabilities of QALIPSIS.

Plugins: 

* Tackle the complexity implied by the multiplicity of technologies and use cases to support in the tool.
* Reduce the coupling between the technical elements, keeping them safer and easier to maintain.

The main advantages of using plugins are:

* A drastic reduction in the footprint of required libraries to run a scenario.
* A better user/developer experience, including:
**  The ability to focus on what is absolutely relevant to the user/developer.
** Having a smaller list of proposals when using auto-completion.
** Faster compilation and start times.
** Smaller artifacts to store, transport and share.

By nature, a plugin is an optional component of QALIPSIS. It allows you to include only what you need while ignoring components and libraries that do not serve your your present concerns.

While a certain number of plugins are proposed by the community to cover the most common use cases and technologies, a plugin can also be an extension only for yourself and remain private.

== Plugin functionality
A plugin contains one or several technical functionalities of the software that are not part of the core. The most common functionalities that a plugin might implement are:

* A step to perform an action, such as:
** Executing requests on a remote system using a specific protocol.
** Polling data from a source.
** Transforming data.
* An events logger.
* A meter registry.

Default implementations of the above functionalities are part of the QALIPSIS core. However, the default implementations may not be suited for your scenario (e.g., if productive testing or constant monitoring are requirements). QALIPSIS plugins enhance the above functionalities by providing related implementations.

== Plugin integration
To create a test scenario, a tiny Gradle project is required. The plugin is simply added as a dependency to the class path of the project. You can read more about scenario configuration within the xref:develop-scenarios/steps.adoc[Step specifications].

If the development of a scenario requires the plugin to compile (when using a step, for example), the dependency to the plugin artifact must be added to the `implementation` dependencies.

If the plugin delivers functionalities that you only need at runtime, add it to the `runtimeOnly` dependencies.

== Using QALIPSIS plugins
QALIPSIS provides several plugins targeting different technologies, systems and protocols.

If you are using the https://github.com/qalipsis/qalipsis-gradle-plugin[Gradle plugin for QALIPSIS], you can add the plugin dependency to your project using the `qalipsisPlugin` configuration. For example:
[source,kotlin,subs="normal"]
----
qalipsis {
    plugins {
        apacheKafka()
        jackson("1.2.3") // You can override the version of the plugin as argument.
    }
}
----

The full list of operators available to add plugins to your project is described in https://github.com/qalipsis/qalipsis-gradle-plugin#adding-qalipsis-plugins[Github].

Alternatively, add the related dependency to your project:

Gradle - Groovy DSL:
[source,groovy,subs="normal"]
----
implementation group:'io.qalipsis.plugin', name: 'qalipsis-plugin-XXX', version: '{revnumber}'
----
Gradle - Kotlin DSL:
[source,kotlin,subs="normal"]
----
implementation("io.qalipsis.plugin:qalipsis-plugin-XXX:{revnumber}")
----

== Plugin configuration

Plugin configuration settings are configured in a .yaml file within a project.

Parameters available for configuration are specific to each plugin and described within each plugin section.

The following sets of parameters are QALIPSIS specific and configured identically across all applicable QALIPSIS plugins: +
* Monitoring +
* Singleton configuration (for Poll and Consume steps)

[[singletonconfiguration-parameters]]
=== Singleton configuration parameters

Singleton configuration parameters configure how polled results are delivered to scenario minions.
The following strategies are available:
Available strategies: `sequential`, `broadcast`, `unicast`, `loop`. +

 * `broadcast`: sends every polled record to all minions. Records are delivered in order, with the oldest buffered record sent first. In this mode, the singleton behaves like a Pub/Sub broadcaster. 

 * `unicast`: sends each polled record to exactly one minion. The singleton acts as a FIFO distributor: Records are stored in a buffer until a minion consumes them. The buffer has a fixed size; when it becomes full, the oldest records are dropped to make space for new ones. 

 * `loop`: sends each polled record to all minions, but once the end of the buffer is reached, it loops back to the beginning. This creates a continuous cyclic stream of the buffered records. 

 * `sequential`: disables singleton execution and runs this step within the parent’s workflow. Data distribution is controlled entirely by the parent step. This mode is typically used when you want to process each record from a source from start to finish once, and then stop publishing until new data is available.
   
 Note::
 Each strategy EXCEPT `sequential` requires `bufferSize` (Int) and `idleTimeout` (Duration); see `singletonConfiguration.bufferSize` and `singletonConfiguration.idleTimeout` definitions below. +


[cols="1,4" options="header"]
|===
| *Parameter* 
| *Description*

| `singletonConfiguration`
a| Configures how polled/consumed results are delivered to scenario minions. +
Specify one of the strategies directly (not wrapped in a lambda). +
*Applicable Steps:* Poll, Consume +
*Optional/Required:* Optional +
*Data Type:* Function +
*Default Value:* unicast(-1, Duration.ZERO) +

.Example
[source,kotlin] 
----
broadcast(bufferSize = 4, idleTimeout = Duration.ofMinutes(1))
----

| `singletonConfiguration.bufferSize`
a| Size of the buffer used to store polled/consumed records before delivery to minions. + 
*Applicable Steps:*  Poll, Consume +
*Optional/Required:* Optional +
*Data Type:* Integer +
*Default Value:* -1 (unlimited buffer size) +

.Example
[source,kotlin] 
----
broadcast(bufferSize = 4, idleTimeout = Duration.ofMinutes(1))
----

| `singletonConfiguration.idleTimeout`
a| Maximum time to wait before delivering polled/consumed records to minions, even if the buffer is not full. +
*Applicable Steps:* Poll, Consume +
*Optional/Required:* Optional +
*Data Type:* java.time.Duration +
*Default Value:* Duration.ZERO +
Note: Duration.ZERO or less signifies no timeout. +

.Example
[source,kotlin] 
---- 
broadcast(bufferSize = 4, idleTimeout = Duration.ofMinutes(1))
----

| `broadcast`
a| Specifies the broadcast parameters for a step. +
*Applicable Steps:* Poll, Consume +
*Optional/Required:* Optional +
*Data Type:* Function +
*Default Value:* N/A +

.Example
[source,kotlin] 
----
broadcast(3, Duration.ofMinutes(1))
----

| `unicast`
a| Specifies the unicast parameters for a step. +
*Applicable Steps:* Poll, Consume +
*Optional/Required:* Optional +
*Data Type:* Function +
*Default Value:* unicast(1, Duration.ZERO) +

.Example
[source,kotlin] 
----
unicast(1, Duration.ofMinutes(3))
----

| `loop`
a| Specifies the loop parameters for a step. +
*Applicable Steps:* Poll, Consume  +
*Optional/Required:* Optional +
*Data Type:* Function +
 *Default Value:* N/A +

.Example
[source,kotlin] 
----
loop(5, Duration.ofMinutes(2))
----

| `sequential`
a| Specifies the sequential parameters for a step. +
*Applicable Steps:* Poll, Consume + 
*Optional/Required:* Optional +
*Data Type:* Function +
*Default Value:* N/A +

.Example
[source,kotlin] 
----
sequential()
----

|===
