== Overview
The RabbitMQ plugin connects QALIPSIS to the https://www.rabbitmq.com/[RabbitMQ] broker.

Technology addressed::
RabbitMQ: https://www.rabbitmq.com/[]

Dependency::
`io.qalipsis.plugin:qalipsis-plugin-rabbitmq`

Namespace in scenario::
`rabbitmq()`

Client library::
RabbitMQ: refer to https://www.rabbitmq.com/[rabbitMQ]

Supported steps::
* <<Consume step,`consume`>>: polls data from the RabbitMQ broker.
* <<Produce step,`produce`>>: pushes data to the RabbitMQ broker.

Examples of usage are provided later in this <<Examples of usage,section>>.

== Examples of usage
=== Consume step
The `consume` step within the RabbitMQ plugin consumes data from queues of the RabbitMQ broker and forwards it to the next step.

This step is commonly used with `join` to assert data or inject data into a workflow.

This step supports concurrent consumers through the `concurrency()` property. When `concurrency` is greater than 1, record order is no longer guaranteed.

To learn more, visit the https://www.rabbitmq.com/#getstarted[RabbitMQ] website.

Ancestor::
Scenario

Functionality::
The `consume` step starts a polling loop for consumption of the configured queue.  As new records arrive, the `consume` step polls them and sends them to the next step.

Example::
[source,kotlin,linenumbers]
----
//Preceded by scenario configuration and a workflow entry point such as `start()`.
 .rabbitmq() <1>
  .consume {
    name = "consume-battery-state" <2>
    connection {
      host = "localhost"
      port = 5672
      username = "Celia45"
      password = "Columbus1963"
    }
    queue("battery_state") <3>
    prefetchCount(10)
    concurrency(1)
  }
  .deserialize(io.qalipsis.api.messaging.deserializer.MessageStringDeserializer::class) <4>
----
<1> Enter into the RabbitMQ namespace.
<2> Set a unique step name for reporting and troubleshooting.
<3> Select the RabbitMQ queue to consume.
<4> Deserialize message bodies as strings using a built-in deserializer.

Tip::
Use the `consume` step with `join` to assert data or inject data into a workflow.

Reference documentation::
* Learn more about `prefetchCount` in the https://www.rabbitmq.com/consumer-prefetch.html[RabbitMQ Prefetch] documentation.
* For `consume` configuration details, see the https://www.rabbitmq.com/documentation.html[RabbitMQ documentation].

//-
=== Produce step
The `produce` step in the RabbitMQ plugin sends messages to the RabbitMQ broker; the results are then forwarded to the next step.

Ancestor::
Scenario

Functionality::
The `produce` step's input and step context are used to generate records that are sent to the broker; results are then forwarded to the next step(s).

Example::
[source,kotlin,linenumbers]
----
//Preceded by scenario configuration and a workflow entry point such as `start()`.
  .rabbitmq() <1>
  .produce { <2>
    connection { <3>
      host = "localhost"
      port = 5672
      username = "qalipsis"
      password = "qalipsis"
    }
    concurrency(1)
    records { stepContext, input -> <4>
      listOf(
        RabbitMqProducerRecord(
          exchange = "battery_state",
          routingKey = "battery_state",
          props = null,
          value = input.toString().toByteArray() <5>
        )
      )
    }
  }
----
<1> Enter into the RabbitMQ namespace to access the `produce` step.
<2> Configure the `produce` step.
<3> Configure the connection to the RabbitMQ broker.
<4> Build the list of records to publish.
<5> Convert the string payload to a `ByteArray`, which is the required type for `RabbitMqProducerRecord.value`.

Reference documentation::
* For more information about RabbitMQ connection configuration, see https://www.rabbitmq.com/api-guide.html#connecting[Connecting to RabbitMQ].
* For more information on configuring the `produce` step, refer to the https://www.rabbitmq.com/documentation.html[RabbitMQ] documentation.

//-
=== RabbitmqProduceAndConsume scenario
Example::
[source,kotlin,linenumbers]
----
@Scenario("rabbitmq-produce-and-consume")
fun rabbitmqProduceAndConsume() {
  scenario { <1>
    minionsCount = 10
    profile {
      immediate()
    }
  }
    .start() <2>
    .jackson()
    .csvToObject(mappingClass = BatteryState::class) { <3>
      classpath(path = "battery-levels.csv")
      header {
        column(name = "deviceId")
        column(name = "timestamp")
        column(name = "batteryLevel").integer()
      }
      unicast()
    }
    .map { it.value } <4>
    .rabbitmq()
    .produce {
      connection {
        host = "localhost"
        port = 5672
        username = "qalipsis"
        password = "qalipsis"
      }
      records { _, input -> <5>
        listOf(
          RabbitMqProducerRecord(
            exchange = "battery_state",
            routingKey = "battery_state",
            props = null,
            value = input.toString().toByteArray() <6>
          )
        )
      }
    }
    .map { it.input } <7>
    .innerJoin()
    .using { produced -> produced.deviceId } <8>
    .on {
      it.rabbitmq()
        .consume {
          name = "consume-battery-state"
          connection {
            host = "localhost"
            port = 5672
            username = "qalipsis"
            password = "qalipsis"
          }
          queue("battery_state") <9>
          prefetchCount(10)
          concurrency(1)
        }
        .deserialize(MessageJsonDeserializer(BatteryState::class)) <10>
        .map { result -> result.value }
        .filterNotNull()
    }
    .having { consumed -> consumed.deviceId } <11>
    .verify { result -> <12>
      result.asClue {
        assertSoftly {
          it.first.batteryLevel shouldBeExactly it.second.batteryLevel
        }
      }
    }

}

----
<1> Configure the scenario with 10 minions and an immediate execution profile.
<2> Start the pipeline; all subsequent steps are chained off this entry point.
<3> Read `battery-levels.csv` from the classpath and deserialize each row into a `BatteryState` instance.
<4> Unwrap the `BatteryState` value from the CSV step output before entering the producer branch.
<5> Build the list of records to publish.
<6> Convert the `BatteryState` to bytes, which is the required type for `RabbitMqProducerRecord.value`.
<7> Extract `RabbitMqProducerResult.input`, the original `BatteryState` from the producer result.
<8> Define the key used to match produced records on the left branch of the join.
<9> Consume records from the same `battery_state` queue that the producer branch writes to.
<10> Deserialize the raw message bytes back into a `BatteryState` instance using the JSON deserializer.
<11> Define the key used to match consumed records on the right branch of the join: `deviceId` on the deserialized `BatteryState`.
<12> Assert that the `batteryLevel` of the produced record exactly matches the `batteryLevel` of the corresponding consumed record for every correlated device.
