= Apache Kafka plugin [.badge.oss]## ##[.badge.commercial]## ##[.badge.cloud]## ##

== Overview
The Apache Kafka plugin connects QALIPSIS to an https://kafka.apache.org/[Apache Kafka] cluster.

Technology addressed::
Apache Kafka: https://kafka.apache.org/[]

Dependency::
`io.qalipsis.plugin:qalipsis-plugin-kafka`

Namespace in scenario::
`kafka()`

Client library::
Apache Kafka Client: refer to https://kafka.apache.org/documentation/#api[Kafka API]

Supported steps::
* <<Consume step,`consume`>>: polls records from Kafka topics and forwards them either by batch or one-by-one.
* <<Produce step,`produce`>>: publishes records to Kafka topics and forwards the original step input.

Examples of usage are provided later in this <<Examples of usage,section>>.

== Configuration
The Kafka Consume and Produce steps are configured in the scenario DSL.

The factory configuration file is used to configure the optional publication of QALIPSIS events and meters to Kafka.

=== Example configuration file
Kafka publication settings are configured in YAML.

[source,yaml,linenumbers]
----
campaign:
  name: kafka-${random.shortuuid}
  configuration:
    validation:
      max-execution-duration: PT40S

logging:
  console: true
  root: INFO

events:
  root: WARN
  export:
    kafka:
      enabled: true
      min-level: INFO
      bootstrap: localhost:9092
      topic: qalipsis-events
      duration-as-nano: false
      linger-period: PT1S
      batch-size: 2000
      format: JSON
      configuration:
        delivery.timeout.ms: 10000

meters:
  export:
    enabled: true
    kafka:
      enabled: true
      bootstrap: localhost:9092
      topic: qalipsis-meters
      prefix: qalipsis
      timestamp-field-name: timestamp
      serializer: json
      linger.ms: 1000
      batch-size: 500
      configuration:
        delivery.timeout.ms: 10000
----

== Properties
This section documents all DSL properties for `consume` and `produce`.

=== Consume properties

=== Shared defaults for Kafka steps
You can define defaults once and let all following Kafka steps inherit them.

[source,kotlin,linenumbers]
----
scenario {
  kafka().defaults { <1>
    bootstrap("localhost:9092")
    properties("security.protocol" to "PLAINTEXT")
  }
}
----
<1> Defaults are applied to subsequent Kafka `consume` and `produce` steps in the same scenario. Individual steps can still override values.

== Examples of usage

=== Consume step
The `consume` step polls Kafka records and forwards them to the next step as batches by default.

Use `.flatten(...)` when you want one consumed result per downstream event.

[source,kotlin,linenumbers]
----
  .kafka()
  .consume { <1>
    name = "orders-consumer"
    bootstrap("localhost:9092")
    topics("orders")
    groupId("orders-group")
    pollTimeout(100)
    offsetReset(OffsetResetStrategy.EARLIEST)
    concurrency(2)
  }
  .flatten(Serdes.Integer().deserializer(), Serdes.String().deserializer()) <2>
  .map { it.record.value } <3>
  .filterNotNull()
----
<1> Configures the consume step and the subscription settings.
<2> Switches output from batch mode to one consumed result at a time, deserializing keys as `Integer` and values as `String` inline.
<3> Reads the deserialized value from `KafkaConsumerResult.record.value`.

=== Produce step
The `produce` step publishes records generated from step context and input.

[source,kotlin,linenumbers]
----
  .kafka()
  .produce( <1>
    Serdes.Integer().serializer(),
    Serdes.String().serializer()
  ) {
    name = "orders-producer"
    bootstrap("localhost:9092")
    clientName("producer-orders") <2>
    records { stepContext, input -> <3>
      listOf(
        KafkaProducerRecord(
          topic = "processed-orders",
          key = input.record.key,
          value = "processed-${input.record.value}"
        )
      )
    }
  }
----
<1> Declares key and value serializers required by `produce(...)`.
<2> Sets the producer client name used by the step.
<3> Builds one or more `KafkaProducerRecord` per consumed input.

=== KafkaProduceAndConsume scenario
This example reads device battery states from a CSV file, publishes each record to a Kafka topic, then immediately consumes from the same topic and correlates the produced and consumed records by device id.

[source,kotlin,linenumbers]
----
@Scenario("kafka-produce-and-consume") // <1>
fun kafkaProduceAndConsume() {
    scenario { // <2>
        minionsCount = 20
        profile {
            immediate() // <3>
        }
    }
    .start()
    .jackson() // <4>
    .csvToObject(mappingClass = BatteryState::class) { // <5>
        classpath(path = "battery-levels.csv") // <6>
        header { // <7>
            column(name = "deviceId")
            column(name = "timestamp")
            column(name = "batteryLevel").integer()
        }
        unicast() // <8>
    }
    .map { it.value } // <9>
    .kafka()
    .produce( // <10>
        keySerializer = Serdes.String().serializer(),
        valueSerializer = Serdes.String().serializer()
    ) {
        bootstrap("localhost:19092") // <11>
        clientName("producer") // <12>
        records { stepContext, input -> // <13>
            listOf(
                KafkaProducerRecord(
                    "battery_state", // <14>
                    value = objectMapper.writeValueAsString(input), // <15>
                    key = input.deviceId // <16>
                )
            )
        }
    }
    .map { it.input } // <17>
    .innerJoin() // <18>
    .using { correlationRecord ->
        correlationRecord.value.deviceId // <19>
    }
    .on {
        it.kafka().consume { // <20>
            bootstrap("localhost:19092")
            topics("battery_state") // <21>
            groupId("kafka-example") // <22>
            offsetReset(OffsetResetStrategy.EARLIEST) // <23>
            pollTimeout(Duration.ofSeconds(1)) // <24>
        }
        .flatten(Serdes.String().deserializer(), jsonSerde<BatteryState>().deserializer()) // <25>
        .map { result ->
            result.record.value // <26>
        }
        .filterNotNull() // <27>
    }
    .having { correlationRecord ->
        correlationRecord.value.deviceId // <28>
    }
    .filterNotNull() // <29>
    .verify { result -> // <30>
        result.asClue {
            assertSoftly {
                it.first.batteryLevel shouldBeExactly it.second.batteryLevel
            }
        }
    }
}
----
<1> Declare the scenario name. QALIPSIS uses this to identify and report the scenario.
<2> Enter the scenario DSL block to configure minions and the execution profile.
<3> Use `immediate()` to start all minions at once with no ramp-up delay.
<4> Enter the Jackson DSL namespace to read data from the CSV file.
<5> Read the CSV file and deserialize each row into a `BatteryState` object.
<6> Load `battery-levels.csv` from the JVM classpath.
<7> Define the column names and types that map to the fields of `BatteryState`. `batteryLevel` is parsed as an integer.
<8> Call `unicast()` so each CSV row is delivered to exactly one minion, preventing duplicate processing.
<9> Unwrap the `DatasourceRecord` and forward the plain `BatteryState` value to the next step.
<10> Start the `produce` step. Declare the key and value serializers as required positional arguments.
<11> Set the bootstrap server address for the Kafka producer.
<12> Set the logical name for this producer client.
<13> Build the list of `KafkaProducerRecord` instances to publish. The lambda receives the step context (`_`, unused) and the `BatteryState` input from the previous step.
<14> Set the target topic name as the first positional argument of `KafkaProducerRecord`.
<15> Serialize the `BatteryState` input to a JSON string and use it as the record value.
<16> Use `deviceId` as the record key so all records for the same device land in the same partition.
<17> Extract the original `BatteryState` input from the produce result to use as the left side of the join.
<18> Start an `innerJoin` to correlate each produced record with a corresponding consumed record.
<19> Use `deviceId` as the correlation key on the left (produce) side of the join.
<20> Start the `consume` step on the right side of the join to poll the same topic the record was published to.
<21> Subscribe to the `battery_state` topic — the same topic the produce step published to.
<22> Set the consumer group id. This consumer manages its own offsets independently from any other consumers.
<23> Set `EARLIEST` as the offset reset strategy to ensure no record produced in this run is missed.
<24> Set the poll timeout to one second to give the broker enough time to make recently produced records available.
<25> Flatten the batch output to one record at a time. Deserialize keys as `String` and values as `BatteryState` using a JSON Serde.
<26> Extract the deserialized `BatteryState` value from the consumed record.
<27> Discard any null values before they reach the join.
<28> Use `deviceId` as the correlation key on the right (consume) side of the join, matching it to the produced record.
<29> Discard any join pair where either side is null, ensuring only fully matched pairs are verified.
<30> Assert that the `batteryLevel` of the produced record exactly matches the `batteryLevel` of the consumed record.
