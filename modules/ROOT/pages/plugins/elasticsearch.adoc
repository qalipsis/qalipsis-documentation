== Overview
The Elasticsearch plugin connects QALIPSIS to https://www.elastic.co/[Elasticsearch] using the elasticsearch REST client.

*Elasticsearch* allows QALIPSIS to poll, save, and analyze data from your system.

This page explains configuration, core concepts, secure connection options, runnable examples and best practices.

Technology addressed::
Elasticsearch: https://www.elastic.co/[]

Dependency::
`io.qalipsis.plugin:qalipsis-plugin-elasticsearch`

Namespace in scenario::
`elasticsearch()`

Client library::
JAVA API:  refer to
https://www.elastic.co/guide/en/elasticsearch/client/java-api-client/8.1/connecting.html[Elasticsearch JAVA Client].

Supported steps::
* <<poll step,`poll`>>: polls the newest data periodically.
* <<save step,`save`>>: saves a batch records generated from the input and step context.
* <<search step,`search`>>: searches records and builds a query from the input and step context.

Example scenarios are provided later in this <<Example scenarios,section>>.

== Example usage

===Poll step

The `poll` step within the Elasticsearch plugin reads batches of documents periodically from Elasticsearch.
The output is a list of `ElasticsearchDocument` containing JSON data or deserialized to objects.

Ancestor::
Scenario

Functionality::
The `poll` step is created only once in a scenario; it is then used throughout the scenario as called.
The `poll` step within the Elasticsearch plugin uses a strategy of “delivered exactly once”.
The output of the `poll` step is an `ElasticsearchDocument` that contains maps of column to values.

Example::
[source,kotlin,linenumbers]
----
elasticsearch().poll {  <1>
    name = "my-poll-step"
    client { RestClient.builder(HttpHost.create("http://localhost:9200")).build() }  <2>
    query {       <3>
        """{
            "query":{
                "match_all":{}
            },
                "sort" : ["_score"]
            }""".trimIndent()
    }
    index("battery-state")   <4>
    pollDelay(Duration.ofSeconds(1))  <5>
}
----
 <1> Start the `poll` step.
 <2> Configure the REST client connection.
 <3> Configure the query.
 <4> Set the `elasticsearch` indices to use for the queries.
 <5> Specifies the delay between query executions.

Tips::
* The `poll` step is generally used in conjunction with a `join` to assert data or inject data into a workflow.
* The `query` parameter requires at least one sorting field to filter the newest results and not fetch the same records repeatedly.
* The output of the `poll` step is an `ElasticsearchDocument` that contains maps of column names to values. If `flatten` is called, the records are provided one by one to the next step; otherwise each poll batch remains complete.

Reference Documentation::
* Refer to
https://www.elastic.co/guide/en/elasticsearch/reference/current/search-search.html#search-search-api-query-params[Elasticsearch Query Parameters] for more information on query parameters.
* Refer to the https://www.elastic.co/guide/en/elasticsearch/reference/current/paginate-search-results.html#search-after[Elasticsearch] documentation for further parameter and configuration information.

== Save step 
The `save` step within the Elasticsearch plugin persists a batch of documents into  https://www.elastic.co/[Elasticsearch].

Ancestor::
Step

Functionality::
The `save` step's input and step context are used to generate a list of documents that are forwarded to https://www.elastic.co/[Elasticsearch].

Example::
[source,kotlin,linenumbers]
----
// `previousStep` is a placeholder for any preceding step (e.g. `collect()` etc.)
previousStep.elasticsearch().save {   <1>
    name = "my-save-step"
    client {    <2>
        RestClient.builder(HttpHost.create("http://localhost:9200"))
            .build() 
    }
    documents { _, input ->  <3>
        listOf(
            Document(
                index = "battery-state",
                id = input.deviceId,
                source = objectMapper.writeValueAsString(input)  
            )
        )
    }
}
----

<1> Start the `save` step.
<2> Configure the REST client connection.
<3> Define the strategy to create documents to save from the input and step context.


Reference Documentation::
Refer to the https://www.elastic.co/guide[Elasticsearch] documentation for further parameter information.

== Search step
The `search` step within the Elasticsearch plugin searches documents in the Elasticsearch database using a search query.

Ancestor::
Step

Functionality::
The `search` step’s input and step context are used to generate a query; the batch of results is then forwarded to the next step(s).

Example::
[source,kotlin,linenumbers]
----
// `previousStep` is a placeholder for any preceding step (e.g. `collect()` etc.)
previousStep.elasticsearch().search {   <1>
    name = "my-search-step"
    client {    <2>
        RestClient.builder(HttpHost.create("http://localhost:9200")).build()
    }
    index { _, _ -> listOf("battery-state") }   <3>
    query { _, input ->     <4>
        """{
            "query": {
                "bool": {
                    "must": [
                        { "match": { "deviceId": "${input.input.deviceId}" }},
                        { "term": { "timestamp": ${input.input.timestamp.epochSecond} }}
                    ]
                }
            }
        }"""
    }  
}
----
<1> Start the `search` step.
<2> Configure the REST client connection.
<3> Set the `elasticsearch` indices to use for the queries.
<4> Configure the builder for the JSON query to perform a search.

Reference Documentation::
* Refer to the https://www.elastic.co/guide[Elasticsearch] documentation for further parameter and configuration information.
* For specifics on `queryParameters` refer to https://www.elastic.co/guide/en/elasticsearch/reference/current/search-search.html#search-search-api-query-paramsElastic[Elasticsearch Query Parameters].
* For specifics on `search` parameters, refer to
 https://www.elastic.co/guide/en/elasticsearch/reference/current/paginate-search-results.html[Elasticsearch Search Parameters].

=== ElasticSearchSaveAndPoll scenario
This example shows a scenario that saves data into Elasticsearch and then polls the table to verify the saved values. It includes a short explanation for each step.

[source,kotlin,linenumbers]
----
@Scenario("elasticsearch-save-and-poll")
fun elasticSearchSaveAndPoll() {

        scenario { <1>
            minionsCount = 10
            profile {
                immediate()
            }
        }
            .start()
            .jackson() <2>
            .csvToObject(mappingClass = BatteryState::class) {

                classpath(path = "battery-levels.csv")
                // we define the header of the csv file
                header { <3>
                    column(name = "deviceId")
                    column(name = "timestamp")
                    column(name = "batteryLevel").integer()
                }
                unicast()
            }
            .map { it.value } <4>
            .elasticsearch() <5>
            .save { <6>
                name = "save"
                client { <7>
                    RestClient.builder(HttpHost.create("http://localhost:9200"))
                        .build() // we create the rest client that will help to submit request to elastic search
                }

                documents { _, input -> <8>
                    listOf(
                        Document(
                            index = "battery-state",
                            id = input.deviceId,
                            source = objectMapper.writeValueAsString(input) // we parse to json the object we want to save
                        )
                    )
                }
            }
            .map { it.input } <9>
            .delay(duration = Duration.ofSeconds(2)) // a small delay of 2 seconds before proceed, because on elastic search persistence of data is not immediate
            .innerJoin()
            .using { correlationRecord -> correlationRecord.value.deviceId } <10>
            .on {
                it.elasticsearch().poll { <11>
                    name = "poll"
                    client { RestClient.builder(HttpHost.create("http://localhost:9200")).build() } <12>
                    query { <13>
                        """{
                                "query":{
                                    "match_all":{}
                                },
                                "sort" : ["_score"]
                            }""".trimIndent()
                    }
                    index("battery-state*") <14>
                    pollDelay(delay = Duration.ofSeconds(1)) <15>
                }
                    .deserialize(targetClass = BatteryState::class) <16>
                    .flatten()
                    .map { record -> <17>
                        record.value
                    }
            }
            .having { correlationRecord -> correlationRecord.value.deviceId } <18>
            .filterNotNull()
            .verify { result -> <19>
                result.asClue {
                    assertSoftly {
                        it.first.batteryLevel shouldBeExactly it.second.batteryLevel
                    }
                }
            }

    }
----
<1> Define the scenario, set the name, number of minions and rampUp.
<2> Start the Jackson step to read data from a CSV file using the csvToObject method to map csv entries to list of utils.BatteryState object.
<3> Define the CSV file header.
<4> Transform the CSV entries to `BatteryState` objects.
<5> Call the Elasticsearch step to save data.
<6> Call the `save` step to persist data into Elasticsearch.
<7> Configure the REST client connection to Elasticsearch.
<8> Define the documents to save in Elasticsearch.
<9> Extract the input for the next step from the save step output.
<10> Start an inner join between the saved data and the polled data using the deviceId as correlation key.
<11> Call the `poll` step to read data from Elasticsearch.
<12> Configure the REST client connection to Elasticsearch.
<13> Define the query to fetch data from Elasticsearch.
<14> Specify the Elasticsearch index to poll data from.
<15> Define the delay between each poll execution.
<16> Deserialize the polled data to `BatteryState` objects.
<17> Extract the `BatteryState` object from the polled record.
<18> Define the correlation key for the join using the deviceId.
<19> Verify that the battery levels from the saved and polled data are identical.
