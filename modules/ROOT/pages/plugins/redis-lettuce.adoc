= Redis Lettuce plugin [.badge.oss]## ##[.badge.commercial]## ##[.badge.cloud]## ##

== Overview
The Redis Lettuce plugin connects QALIPSIS to https://redis.io/[Redis] using the https://lettuce.io/[Lettuce] Java client.

Technology addressed::
Redis: https://redis.io/[]

Dependency::
`io.qalipsis.plugin:qalipsis-plugin-redis-lettuce`

Namespace in scenario::
`redisLettuce()`

Client library::
Lettuce Core: refer to https://lettuce.io/[Lettuce].

Supported steps::
* Core Redis model:
** <<Poll step,`pollScan`, `pollSscan`, `pollHscan`, `pollZscan`>>: polls Redis keys or structures periodically.
** <<Save step,`save`>>: writes records to Redis with `SET`, `SADD`, `HSET`, or `ZADD`.
* Redis Streams model:
** <<Consume step,`streamsConsume`>>: consumes records from a Redis stream with a consumer group.
** <<Produce step,`streamsProduce`>>: pushes records to a Redis stream.

Model separation::
* Core Redis (`poll*`, `save`) targets keys and Redis data structures.
* Redis Streams (`streamsConsume`, `streamsProduce`) targets stream entries and consumer groups.

Examples of usage are provided later in this <<Examples of usage,section>>.

== Configuration
The redis-lettuce steps are configured in the scenario DSL.

== Configuration

=== Example configuration file
Redis Lettuce configuration settings are configured in an `redis-lettuce.yaml` file.

[source,yaml,linenumbers]
----
campaign:
  name: redis-lettuce-${random.shortuuid}
  configuration:
    validation:
      max-execution-duration: PT40S

events:
  root: WARN

report:
  export:
    console-live:
      enabled: false
----

=== Configuration parameters
Available parameters are described in the table below.

[cols="1,4" options="header"]
|===
| *Parameter*
| *Description*

a| Registers default Redis connection and monitoring for subsequent Redis Lettuce steps in the same scenario or on the same branch. +

scenario {

*Optional/Required:* Optional +
*Data Type:* `List<String>` +

*Applicable Steps:* Poll, Save, Consume, Produce +
*Optional/Required:* Optional +

a| Redis connection topology. +
*Applicable Steps:* Poll, Save, Consume, Produce +

*Data Type:* `RedisConnectionType` (enum) one of `SINGLE`, `CLUSTER`, `SENTINEL` +

*Applicable Steps:* Poll, Save, Consume, Produce +
*Optional/Required:* Optional +

*Optional/Required:* Optional +
*Data Type:* `String` +
*Default Value:* `""` +

| `connection.masterId`
a| Redis Sentinel master identifier. Required when `connection.redisConnectionType = SENTINEL`. +
*Applicable Steps:* Poll, Save, Consume, Produce +
*Optional/Required:* Optional in general, required for `SENTINEL` +
*Data Type:* `String` +
*Default Value:* `""` +

[source,kotlin]
----

    nodes = listOf("localhost:26379")

=== B. Core Redis Model Properties (Poll / Save)

[cols="1,4" options="header"]
|===
| *Property*
| *Description*

| `keyOrPattern`
a| Defines the target scanned by polling. +
For `pollScan`, this is a key pattern. +
For `pollSscan`, `pollHscan`, and `pollZscan`, this is the key to scan. +
*Applicable Steps:* Poll +
*Optional/Required:* Required (`@NotBlank`) +
*Data Type:* `String` +
*Default Value:* `""` +

.Example
[source,kotlin]
----
keyOrPattern("battery_state_*")
----

| `pollDelay`
a| Delay between two poll executions. +
*Applicable Steps:* Poll +
*Optional/Required:* Optional (`@PositiveOrZeroDuration`) +
*Data Type:* Function overload accepting `Duration` or `Long` +
*Default Value:* `Duration.ofSeconds(10)` +

.Example
[source,kotlin]
----
pollDelay(Duration.ofSeconds(1))
// or
pollDelay(1000L)
----

| `records`
a| Generates records to persist for each incoming input. +
Supported record types: `ValueRecord` (`SET`), `SetRecord` (`SADD`), `HashRecord` (`HSET`), `SortedRecord` (`ZADD`). +
*Applicable Steps:* Save +
*Optional/Required:* Required to effectively write data +
*Data Type:* Lambda which takes the stepContext and the input and returns a list of `LettuceSaveRecord` +
*Default Value:* `{ _, _ -> emptyList() }` +

.Example
[source,kotlin]
----
records { stepContext, input ->
    listOf(
        ValueRecord("device:42", "ok"),
        SetRecord("devices", "device:42"),
        HashRecord("device:42:state", mapOf("batteryLevel" to "98")),
        SortedRecord("device:42:history", 1712520900.0 to "98")
    )
}
----

[cols="1,4" options="header"]
|===

== Examples of usage

=== A. Core Redis examples

==== Poll step
The `poll` family in the Redis Lettuce plugin periodically scans Redis and forwards new records.

Ancestor::
Scenario

Functionality::
Use exactly one scan flavor depending on the targeted Redis structure:
* `pollScan`: key pattern scan (`SCAN`)
* `pollSscan`: set scan (`SSCAN`)
* `pollHscan`: hash scan (`HSCAN`)
* `pollZscan`: sorted set scan (`ZSCAN`)

Example (`pollSscan`)::
+
[source,kotlin,linenumbers]
----
redisLettuce().pollSscan {
    name = "poll-sscan" <1>
    connection { <2>
        nodes = listOf("localhost:6379")
        database = 0
        redisConnectionType = RedisConnectionType.SINGLE
    }
    keyOrPattern("battery-state-devices") <3>
    pollDelay(Duration.ofSeconds(1)) <4>
    monitoring {
        events = true
        meters = true
    }
}.flatten() <5>
----
<1> Creates a poll step using the Redis `SSCAN` command.
<2> Configures the Redis connection for this step.
<3> Defines the Redis key to scan for `SSCAN`.
<4> Sets the delay between two poll executions.
<5> Emits one `RedisRecord<String>` at a time instead of batched `LettucePollResult<String>`.

Tips::
* Use `pollScan` when you need wildcard key discovery.
* Use `pollSscan`, `pollHscan`, or `pollZscan` when you target a known key of a specific Redis structure.

Reference Documentation::
Refer to https://redis.io/docs/[Redis] and https://lettuce.io/core/release/reference/[Lettuce] for command and topology details.

==== Save step
The `save` step writes generated records to Redis.

Ancestor::
Step

Functionality::
For each incoming input, `records {}` generates one or more `LettuceSaveRecord` objects. The record type selects the Redis command.

Example::
+
[source,kotlin,linenumbers]
----
.redisLettuce()
.save {
    name = "save" <1>
    connection { <2>
        nodes = listOf("localhost:6379")
        database = 0
        redisConnectionType = RedisConnectionType.SINGLE
    }
    records { stepContext, input -> <3>
        listOf(
            HashRecord(
                key = "battery:${input.deviceId}",
                value = mapOf(
                    "deviceId" to input.deviceId,
                    "batteryLevel" to input.batteryLevel.toString(),
                    "timestamp" to input.timestamp.toString()
                )
            )
        )
    }
}
----
<1> Creates a save step.
<2> Configures the Redis connection for this step.
<3> Builds `HSET` records (via `HashRecord`) from the current input.

Tips::
* Use `ValueRecord` for `SET`, `SetRecord` for `SADD`, `HashRecord` for `HSET`, and `SortedRecord` for `ZADD`.
* Use a `collect` step before `save` if you need to reduce command churn and write in larger logical batches.

Reference Documentation::
Refer to https://redis.io/docs/[Redis] and https://lettuce.io/core/release/reference/[Lettuce] for command and topology details.

=== B. Redis Streams examples

==== Consume step
The `streamsConsume` step consumes records from a Redis stream using a consumer group.

Ancestor::
Scenario

Functionality::
The step reads from a stream key and consumer group and can emit data as batches or single records.

Example::
+
[source,kotlin,linenumbers]
----
redisLettuce().streamsConsume {
    name = "consume" <1>
    connection { <2>
        nodes = listOf("localhost:6379")
        database = 0
        redisConnectionType = RedisConnectionType.SINGLE
    }
    streamKey("battery-state-stream") <3>
    group("battery-consumers") <4>
    offset(LettuceStreamsConsumerOffset.FROM_BEGINNING) <5>
    concurrency(3) <6>
}.flatten() <7>
----
<1> Creates a stream consumer step.
<2> Configures the Redis connection for this step.
<3> Defines the stream to read.
<4> Defines the consumer group name.
<5> Starts from the beginning when no known state exists for the group.
<6> Runs three concurrent stream consumers.
<7> Emits one `LettuceStreamsConsumedRecord` at a time.

Tip::
Use `offset(LettuceStreamsConsumerOffset.LAST_CONSUMED)` when you want the group to continue from pending/acked state and `LATEST` when you only want new records.

Reference Documentation::
Refer to https://redis.io/docs/[Redis] and https://lettuce.io/core/release/reference/[Lettuce] for stream and topology details.

==== RedisSaveAndPoll scenario
[source,kotlin,linenumbers]
----
@Scenario("redis-save-and-poll") // <1>
    fun redisSaveAndPoll() {

        scenario { // <2>
            minionsCount = 20 // <3>
            profile {
                immediate() // <4>
            }
        }
            .start() // <5>
            .jackson() // <6>
            .csvToObject(mappingClass = BatteryState::class) { // <7>

                classpath(path = "battery-levels.csv") // <8>
                // we define the header of the csv file
                header { // <9>
                    column(name = "device_id")
                    column(name = "timestamp")
                    column(name = "battery_level").integer()
                }
                unicast() // <10>
            }
            .map {
                it.value // <11>
            }
            .redisLettuce() // <12>
            .save { // <13>
                connection { // <14>
                    nodes = listOf("localhost:6379")
                    database = 0
                    redisConnectionType = RedisConnectionType.SINGLE
                    authPassword = ""
                    authUser = ""
                }

                records { stepContext, input -> // <15>
                    listOf(
                        SetRecord( // <16>
                            "battery_state_save_and_poll",
                            objectMapper.writeValueAsString(input) // <17>
                        )
                    )
                }
            }
            .map {
                it.input // <18>
            }
            .innerJoin() // <19>
            .using { correlationRecord -> correlationRecord.value.deviceId } // <20>
            .on { // <21>
                it.redisLettuce()
                    .pollSscan { // <22>
                        connection { // <23>
                            nodes = listOf("localhost:6379")
                            database = 0
                            redisConnectionType = RedisConnectionType.SINGLE
                            authPassword = ""
                            authUser = ""
                        }

                        keyOrPattern("battery_state_save_and_poll") // <24>

                        pollDelay(Duration.ofSeconds(1)) // <25>
                    }
                    .flatten() // <26>
                    .map { result ->
                        val batteryState = objectMapper.readValue(result.value, BatteryState::class.java) // <27>
                        batteryState
                    }
            }
            .having { correlationRecord -> correlationRecord.value.deviceId } // <28>
            .filterNotNull() // <29>
            .verify { result -> // <30>
                result.asClue {
                    assertSoftly {
                        it.first.batteryLevel shouldBeExactly it.second.batteryLevel // <31>
                    }
                }
            }

    }
----
<1> Register this function as the `redis-save-and-poll` scenario.
<2> Define the scenario with 20 minions launched immediately using the `immediate()` profile.
<3> Set the number of concurrent minions to 20.
<4> Launch all minions simultaneously with no ramp-up delay.
<5> Mark the start of the step chain.
<6> Initialize the Jackson step to enable CSV file reading with Jackson deserialization.
<7> Map each CSV row to a `BatteryState` instance using `csvToObject`.
<8> Load the CSV file from the classpath at the given path.
<9> Declare the CSV column names and types. `battery_level` is parsed as an integer.
<10> Call `unicast()` to ensure each CSV record is consumed by exactly one minion.
<11> Unwrap the `CsvRecord` envelope to obtain the raw `BatteryState` value for the next step.
<12> Enter the Redis Lettuce plugin namespace. Required before any Redis Lettuce step can be chained.
<13> Save each `BatteryState` as a Redis set member. The step output is wrapped in `LettuceSaveResult<BatteryState>`.
<14> Configure the Redis connection for the save step. Targets a standalone node on `localhost:6379`, database `0`, with no authentication.
<15> Build the list of Redis records to write for each input. The step context is unused (`_`); only `input` is needed.
<16> Use `SetRecord` to issue a Redis `SADD` command, adding the value to the named set.
<17> Serialize the `BatteryState` input to JSON before storing it in the Redis set.
<18> Unwrap the `LettuceSaveResult` to extract the original `BatteryState` input and pass it to the next step.
<19> Correlate the save output with the poll output using `innerJoin`.
<20> Use `deviceId` as the join key on the left (save) branch.
<21> Define the right branch of the join. `it.redisLettuce()` enters the Redis Lettuce namespace from the scenario context.
<22> Start the poll step using `SSCAN` to periodically scan members of a Redis set.
<23> Configure the Redis connection for the poll step.
<24> Set the Redis key to scan. Must match the key used in the `SetRecord` on the save side.
<25> Wait one second between poll executions.
<26> Emit each `RedisRecord<String>` individually rather than as a batched `LettucePollResult<String>`.
<27> Deserialize each raw JSON string back into a `BatteryState` for comparison.
<28> Use `deviceId` as the join key on the right (poll) branch. Must match the key declared in `.using {}`.
<29> Discard pairs where either side returned `null`.
<30> Assert the saved and retrieved values match. `asClue` attaches the full result to any assertion failure message.
<31> Assert that the `batteryLevel` read back from Redis is exactly equal to the value that was saved.

Reference Documentation::
Refer to https://redis.io/docs/[Redis] and https://lettuce.io/core/release/reference/[Lettuce] for stream and topology details.
